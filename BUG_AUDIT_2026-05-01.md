# Stuart AI Comprehensive Bug Identification Audit

**Date:** 2026-05-01  
**Scope:** Whole repository static + dynamic validation, with emphasis on stated product promises vs delivered implementation.  
**Mode:** Identification only (no fixes).

---

## 1) Executive Summary

This codebase currently has **systemic breakage** at import/test-collection level, plus feature-integrity gaps around the flagship **Stealth Mode** and partial MCP plumbing.

Top conclusions:

1. **Test suite is not runnable in current repo state** due to missing modules, broken import paths, and unresolved dependency assumptions.
2. **Core package import side effects are too heavy** (`core/__init__.py` eagerly imports runtime modules that depend on unavailable infra), causing cascading failures.
3. **Multiple tests reference files/modules that do not exist** (e.g., `tests/hybrid_planner.py`, `automation.obsidian_sync`, `cognitive.hybrid_planner`, `cognitive.prompt_manager`).
4. **Dependency contract is inconsistent**: code imports packages that are not guaranteed in env and not guarded by optional-loading patterns.
5. **Stealth mode “cause-of-failure” likely includes UX/API mismatch**: frontend `setStealthMode()` is CSS-only and does not trigger Windows stealth primitives, while docs/README describe stronger behavior.
6. **MCP bridge implementation is functional baseline but fragile** for full MCP compatibility (message framing/protocol assumptions are narrower than many MCP servers).

## 2) Methodology

- Ran full collection test pass using `pytest -q`.
- Performed focused source inspection on failures and stealth/MCP paths.
- Mapped observed behavior against product claims in README/docs.

## 3) Bug Inventory (Granular)

## A. Critical Reliability / Build Integrity

### A1. Hard import dependency on missing `database` module in runtime path
- `core/agent_runtime.py` imports `from database import get_db_connection`.
- `core/__init__.py` eagerly imports `agent_runtime`, causing **any** `import core...` usage to fail when `database` module is absent.
- Impact: Massive blast radius across tests and runtime startup paths.

**Evidence**
- `core/agent_runtime.py` line 17 import statement.
- `core/__init__.py` line 7 eager import.
- `pytest` output shows repeated `ModuleNotFoundError: No module named 'database'` during collection.

### A2. Package initialization anti-pattern: eager heavyweight imports
- `core/__init__.py` imports orchestrator/runtime symbols at import time.
- This turns a lightweight namespace import into infra-dependent execution.
- Impact: Prevents partial module usage and breaks test isolation.

### A3. Missing internal modules referenced by tests
- `tests/test_hybrid_planner.py` imports `cognitive.hybrid_planner` (missing).
- `tests/test_llm_planning.py` and `tests/test_rule_based_planning.py` also import same missing module.
- `tests/test_prompt_manager.py` imports `cognitive.prompt_manager` (missing).
- `tests/test_obsidian_sync.py` imports `automation.obsidian_sync` (missing).

### A4. Tests dynamically load non-existent files
- `tests/test_executor.py` and `tests/test_tasks_7_4_to_7_9.py` attempt loading `tests/hybrid_planner.py` via `spec_from_file_location`, but file is absent.
- Impact: deterministic collection failure.

### A5. Incorrect relative imports inside tests package
- `tests/test_config.py` imports `.system_config` (missing).
- `tests/test_event_bus.py` imports `.event_types` and `.event_bus` from tests package paths that do not exist there.

## C. Stealth Mode vs Claimed Behavior (Feature-Integrity Bugs)

### C1. Broken documentation link path
- README links to `[PROCTORING_STEALTH_GUIDE.md](PROCTORING_STEALTH_GUIDE.md)` at root, but actual file is under `docs/PROCTORING_STEALTH_GUIDE.md`.
- Impact: users cannot reach full stealth instructions from advertised link.

### C2. Frontend stealth preset is visual-only (possible root cause of “stealth broken” perception)
- `web/js/live-controls.js` defines `setStealthMode()` that only changes CSS transparency values.
- It does **not** invoke Windows-level protections (`SetWindowDisplayAffinity`, taskbar hiding, click-through) from `window_manager.py`.
- Impact: user may expect “proctoring stealth mode” but only receives opacity changes unless separate backend hotkey path is triggered.
- This is a likely mismatch when comparing expected stealth delivery vs actual delivered behavior in web controls.

### C3. OS dependency mismatch risk for stealth promises
- README markets stealth as key capability, but implementation relies heavily on Win32 behavior (`window_manager.py`); non-Windows paths degrade/disable features.
- If not communicated at runtime in all entrypoints, users may assume parity where none exists.

## D. MCP Bridge Robustness Gaps

### D1. Narrow protocol framing assumption
- `tools/mcp_client.py` reads/writes newline-delimited JSON.
- Many MCP transports use structured framing conventions depending on implementation details; strict line-based assumptions can fail with alternative server wrappers.
- Potential symptom: intermittent parse/timeout/handshake failures against some MCP servers.

### D2. Silent degradation on tool errors
- `call_tool()` returns stringified `[Tool Execution Error] ...` rather than structured error object and does not propagate detailed metadata.
- Impact: upstream planner/agent may mis-handle failures as plain text success payload.

### D3. Resource lifecycle risk
- `shutdown()` uses `terminate()` but no `wait()`/timeout/kill fallback; potential zombie risk under edge conditions.

## E. Test Quality / Delivery Drift Bugs

### E1. Test class not collected due to constructor
- Pytest warning indicates test class with `__init__` (`tests/test_reflection_enhanced.py`) cannot be collected.
- Impact: false confidence (tests exist but do not run).

### E2. Legacy/fragmented test architecture
- Mixed `unittest` and `pytest` styles with ad-hoc `sys.modules` manipulation and file-loader hacks indicates brittle test design and high maintenance burden.

## 4) Root-Cause Themes

## 6) “Stealth mode broken” + Aura comparison note

Using only repository-local evidence, the most probable local cause is **control-plane split**:
- Web UI “stealth mode” = CSS preset.
- Actual anti-capture/click-through/taskbar cloaking = native Windows manager path + global hotkeys.

If Aura AI is perceived as better in stealth consistency, this repo likely loses on:
1. Single “stealth” command not uniformly mapped to full native stack.
2. Ambiguous naming (`setStealthMode` in frontend implies full stealth but is only visual).
3. No explicit runtime state verification surfaced in UI that confirms native stealth primitives were actually applied.
